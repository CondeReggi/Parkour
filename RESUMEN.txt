================================================================================
PARKOUR APP — RESUMEN DEL ESTADO ACTUAL
================================================================================

App de escritorio (Electron + React) local-first para aprender parkour de forma
estructurada y segura. MVP sin análisis IA: registro, evaluación, recomendación
de rutinas/movimientos, revisión personal de videos, sistema completo de
gamificación (XP, niveles, misiones, logros, rachas inteligentes) y skill-tree
visual. Toda la persistencia es local en SQLite vía Prisma.

Versión: 0.1.0
Idioma de la UI: español (rioplatense).

================================================================================
1. STACK Y DECISIONES DE PROYECTO
================================================================================

Runtime y bundling
- Electron 33 + electron-vite (HMR del renderer, rebuild del main en cambios).
- Tres procesos con bundles separados: main, preload, renderer.
- TypeScript estricto en todos lados (web + node configs separadas).

Frontend
- React 18 + react-router-dom 7 (HashRouter — file:// en producción).
- TanStack Query 5 para data fetching y cache.
- React Hook Form + Zod (resolver) para forms y validación.
- Tailwind CSS 3 + shadcn/ui style "new-york" baseColor "zinc".
- lucide-react para íconos.
- Zustand instalado (no usado activamente todavía).
- NINGUNA librería de gráficos: todos los charts son SVG/divs puros.
- NINGUNA librería de modales (Radix Dialog NO está instalado): hay un Dialog
  minimalista propio en components/ui/dialog.tsx.
- NINGUNA librería de grafos: el skill-tree usa CSS grid + SVG superpuesto
  con paths cubic-bezier calculados via getBoundingClientRect.

Backend (proceso main)
- Prisma 6 con SQLite. El archivo .db vive en app.getPath('userData')/parkour.db
  (resuelto en src/main/db/client.ts y forzado por app.setName('parkour-app')).
- IPC tipado: el contrato vive en src/shared/api.ts; preload lo implementa
  invocando channels '<feature>:<method>'; main lo registra con ipcMain.handle.
- Zod en el main valida cada payload IPC antes de tocar el repo.

Aliases TS (electron.vite.config.ts)
- @           → src/renderer
- @shared     → src/shared
- (sólo renderer usa @; main y preload sólo @shared)

10. SISTEMA DE XP Y NIVELES
================================================================================

Tabla: XpEvent (ver §4). Cada acción se traduce a un XpEvent con un source
y un sourceRefId; el unique garantiza que esa acción específica nunca
otorgue XP dos veces.

Recompensas (XP_REWARDS):
  session_finalized:        50
  movement_practicing:      10
  movement_mastered:        30
  video_uploaded:            5
  video_reviewed:           10
  spot_registered:          15
  quest_claimed:             0  (variable: amount viene de la quest)
  achievement_unlocked:      0  (variable: amount viene del catálogo)

Curva de niveles (services/xpLevels.ts):
  threshold(N) = 50 * N * (N - 1)
    N1=0, N2=100, N3=300, N4=600, N5=1000, N6=1500, ...
  Incremento por nivel: +100, +200, +300, ... (regular y predecible).

grantForActive(source, sourceRefId): Promise<boolean>
  Devuelve TRUE si el XpEvent fue nuevo. Los repos usan ese boolean para
  decidir si propagan progreso a misiones. Idempotencia garantizada por
  unique.

grantForActiveWithAmount(source, sourceRefId, amount): variante para
  fuentes con recompensa variable (quest_claimed, achievement_unlocked).

Backfill lazy: la primera vez que se pide el estado, si count==0, recorre
sessions/progress/videos/spots y los traduce a XpEvents. Idempotente: si se
ejecuta dos veces, grantOnce rechaza duplicados.

UI:
  - LevelProgressCard (dashboard): nivel grande, XP totales, barra al
    próximo nivel, link a /progress.
  - XpBreakdownCard (/progress): tarifa por fuente + cuánto llevás de
    cada una. quest_claimed y achievement_unlocked se muestran como
    "Recompensa variable".

13. SISTEMA DE RACHAS INTELIGENTES
================================================================================

Función pura: services/streaks.ts → computeStreakState(sessions, recoveries).

Clasificación por día (classifyDay):
  - Sesión finalizada en ese día        → 'training'
  - DailyActivity active_recovery       → 'active_recovery'
  - Ninguna actividad, pero última      → 'justified_rest'
    sesión previa tuvo painAfter >= 7
    o fatigueAfter >= 7 y el gap es
    1-2 días
  - Else                                → 'rest'

Racha actual:
  - Si hoy es activo (training/recovery), arranca contando desde hoy.
  - Si hoy no hay nada y ayer sí fue activo, arranca desde ayer
    (tolerancia 1 día).
  - Else → 0.
  - Retrocede mientras el día sea training/recovery/justified_rest.

Mejor racha histórica: corre running counter desde el primer día con
actividad; reset en cada 'rest' puro.

Días activos esta semana (lunes-domingo): cuenta con training o recovery.

Recomendación (pickRecommendation) tipo coach, en rioplatense:
  - Nunca entrenó            → invita a empezar
  - Hoy ya entrenó           → cerrá bien, cuidá la recuperación
  - Hoy hizo recovery        → la racha sigue viva
  - Hoy es justified_rest    → sugiere recuperación activa (suggest=true)
  - Última sesión con pain/  → sugiere recuperación activa (suggest=true)
    fatiga >= 7
  - 3+ días seguidos de      → sugiere recuperación activa
    training puro
  - Idle con racha viva      → "no la dejes pasar"
  - Racha rota               → "una sesión cortita y empezás racha nueva"

markActiveRecovery: upsert idempotente por (profileId, date, type='active_recovery').
Dispara achievementRepository.evaluateAndUnlockForActive() después.

UI:
  - StreakCard (dashboard): racha actual + mejor + estado de hoy con ícono +
    alert con recomendación + botón "Marcar recuperación activa" cuando
    corresponde.

19. PATRONES DE CÓDIGO RECURRENTES
================================================================================

Hook pattern (TanStack Query):
  export const xxxKeys = {
    all: ['xxx'] as const,
    list: () => [...xxxKeys.all, 'list'] as const,
    byId: (id) => [...xxxKeys.all, 'byId', id] as const
  }
  // Mutations invalidan SIEMPRE en cascada lo que pueda haber cambiado:
  onSuccess: () => {
    void qc.invalidateQueries({ queryKey: xxxKeys.all })
    void qc.invalidateQueries({ queryKey: gamificationKeys.all })
    void qc.invalidateQueries({ queryKey: questsKeys.all })
    void qc.invalidateQueries({ queryKey: achievementsKeys.all })
    void qc.invalidateQueries({ queryKey: streakKeys.all })
  }

Invalidación cruzada (la "regla de oro"):
  Cada mutation que el repo del main convierte en XP/quest/achievement/streak
  tiene que invalidar las query keys correspondientes en el frontend.
  Tabla de quién invalida qué:
    useFinalizeSession      → sessions, gamification, quests, achievements, streak
    useSetMovementProgress  → movements, gamification, quests, achievements
    useCreateVideo          → videos, gamification, quests, achievements
    useUpdateVideo          → videos, gamification, quests, achievements
    useCreateSpot           → spots, gamification, quests, achievements
    useClaimQuest           → quests, gamification, achievements
    useMarkActiveRecovery   → streak, achievements, sessionsKeys.stats()

IPC pattern:
  - Channel: 'feature:method'.
  - Handler: parsea con Zod → llama repo → devuelve DTO plano.
  - preload: invoke<T>(channel, payload) tipado por ParkourApi.

grantForActive booleano:
  Toda acción que otorga XP también potencialmente progresa misiones. El
  patrón es:
    const granted = await xpEventRepository.grantForActive(source, refId)
    if (granted) {
      await questRepository.progressForActive(metric)
      // …otros progresos relacionados
    }
    await achievementRepository.evaluateAndUnlockForActive()
  La evaluación de achievements se llama SIEMPRE (es idempotente), pero
  fuera del if (granted) para que también cubra cambios indirectos.

Idempotencia con createMany (SQLite):
  - SQLite no soporta `skipDuplicates`.
  - Patrón: SELECT upfront los que ya existen → filtrar el array →
    createMany del resto. Como main es single-process, no hay race.

Forms:
  - React Hook Form + zodResolver(zSchema).
  - Inputs nullables: value={field.value ?? ''} + onChange convierte ''→null.
  - Selects con valor "sin X": constante '__none__' como SelectItem value.

Repos:
  - Una sola fuente de DTOs: el repo define toDto(row) y todo lo pasa por ahí.
  - Parsing de JSON-strings tolerante (try/catch).
  - Includes con Prisma.XxxInclude tipados, payloads narrow con
    Prisma.XxxGetPayload<{ include: typeof X_INCLUDE }>.

Diálogos modales:
  - useState con union discriminada para el estado del modal:
      type DialogState = { kind:'closed' } | { kind:'edit', item } | …
  - Sólo un Dialog por página; switch interno por kind.

Funciones puras compartidas:
  - learningPathStatus.ts es la fuente de verdad de 4 estados de movement
    (locked / available / practicing / mastered).
  - heroQuote.ts, pickGuidedRoutine.ts, streaks.ts, xpLevels.ts, etc.
    todas funciones puras testeables sin Prisma ni Electron.

- PhaseTodoCard.tsx ya no se usa. Queda como template reutilizable.

  borrarse o moverse a /progress.

  del coach guiado (lo reemplazó GuidedSessionView + FeedbackStep).

  sólo entrenamientos). La racha "inteligente" vive sólo en StreakCard.

  import — para MVP local-first no aplica.
- safetyOverridden existe en el modelo pero la UI del coach guiado no lo
